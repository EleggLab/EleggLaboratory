# OpenClaw / 바이브 코딩 개발 리서치 (2026-03-02)

## 1) 목적
이 문서는 OpenClaw(메신저 기반 에이전트 개발)와 바이브 코딩(대화형 AI 개발)으로 작업할 때의 **제한 사항**, **실전 예시**, **결과물 형태**를 빠르게 참고할 수 있도록 정리한 리서치 노트다.

---

## 2) 핵심 요약

- OpenClaw는 "메신저 ↔ 도구(파일/터미널/브라우저/노드)"를 연결하는 개인형 게이트웨이 구조다.
- 바이브 코딩은 속도/탐색에 강하지만, 재현성/안정성/보안 측면에서 운영 규칙이 필요하다.
- 실제 프로젝트 완료 품질은
  1) 권한/인증,
  2) 실행 환경(로컬 GUI/원격),
  3) 검증 자동화(test/build)
  4) Git 운영(커밋/원격 push)
  를 어떻게 관리하느냐에 크게 좌우된다.

---

## 3) OpenClaw 기반 개발의 제한 사항

### 3.1 도구/정책 제한
- 사용 가능한 도구는 세션의 정책에 의해 제한된다(예: web_search API 키 필요, 특정 툴 deny 가능).
- 일반적으로 파일/셸/프로세스/메시징/브라우저/노드 도구를 사용하지만, 조직별 보안 정책에 따라 불가할 수 있다.

### 3.2 인증/권한 병목
- GitHub push, 채널 로그인, 외부 API 호출은 토큰/자격증명 없으면 막힌다.
- 특히 `git push`는 로컬 credential helper/gh/PAT 부재 시 실패가 잦다.

### 3.3 UI 확인 한계 (원격 대화 vs 설치된 머신)
- 에이전트가 앱을 실행해도 **실행 창은 설치된 머신**에 뜬다.
- 다른 데스크톱/다른 기기에서 대화 중이면 창 자체를 직접 보기 어렵고, 로그(`flutter run`, VM service URL 등)로 간접 검증해야 한다.

### 3.4 장시간 작업/비동기 작업 관리
- 설치/빌드/테스트는 오래 걸리므로 백그라운드 프로세스 관리(`exec + process poll`)가 필수다.
- 무한 폴링/반복 호출 루프를 피해야 한다.

### 3.5 보안 모델 한계
- OpenClaw 보안 가이드는 "개인/신뢰 경계" 모델을 전제로 한다.
- 적대적 멀티테넌트(서로 불신 사용자 다수 공유) 보안 경계로 쓰려면 게이트웨이/호스트를 분리해야 한다.

### 3.6 정보 수집 제약
- `web_search`는 Brave API 키가 없으면 검색 불가.
- 이 경우 `web_fetch` + 공식 문서 URL 직접 접근 방식으로 우회 가능하지만, 탐색 범위는 줄어든다.

## 4) 바이브 코딩 방식의 제한 사항

### 4.1 장점
- 아이디어 → 프로토타입 전환 속도가 빠르다.
- 반복 개선(요구 → 수정 → 실행 → 피드백)에 강하다.

### 4.2 단점
- 코드 일관성/아키텍처 품질이 흔들릴 수 있다.
- 한 번에 많은 변경을 하면 테스트 부채가 생긴다.
- "돌아가면 됨" 상태로 끝나기 쉬워 운영 단계에서 품질 이슈가 발생한다.

### 4.3 운영 리스크
- 비밀값(토큰) 채팅 노출
- 검증 없이 커밋/배포
- 로컬 성공과 실환경 실패 간 괴리

## 5) 실전 예시 (이번 워크플로우 기준)

### 예시 A) 개발 환경 부트스트랩
- 설치 확인: git/node/python/flutter/android sdk 등 존재 여부 점검
- 부족한 항목 설치: JDK, Flutter, Android cmdline-tools, 라이선스 승인
- 확인: `flutter doctor -v`

**산출물**
- Android toolchain usable
- Flutter/Dart usable
- PATH/SDK 환경변수 반영

### 예시 B) 소형 프로젝트 기반 바이브 코딩
- 소스 확보: `flutter/samples` 클론
- 대상 선정: `form_app` (작고 구조 명확)
- 변경: 로그인 UX 개선(검증/강도바/remember me/로딩 상태)
- 검증: `flutter analyze`, `flutter test`, `flutter run -d linux`

**산출물**
- 기능 변경 코드
- 테스트 통과
- 실행 가능 로그

### 예시 C) Git 운영
- `user.name / user.email` 미설정 시 커밋 실패
- 설정 후 커밋 성공
- remote 미설정/인증 미완료 시 push 실패
- remote 추가 + PAT 인증 후 push 성공

**산출물**
- 원격 저장소 반영 완료(main tracking)

## 10) 결론

OpenClaw + 바이브 코딩 조합은 "빠른 구현"에는 매우 강력하다.
다만 실무 품질을 얻으려면 **권한/인증 관리**, **원격 실행 가시성 보완(로그 중심 검증)**, **테스트/커밋 규율**을 반드시 같이 운영해야 한다.

즉, 속도는 바이브 코딩으로 얻고, 신뢰성은 체크리스트와 Git/테스트 습관으로 보완하는 방식이 가장 현실적이다.
